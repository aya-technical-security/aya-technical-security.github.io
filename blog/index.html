2026.04.24 Web制作

WordPressなしで作る、和風お手紙ブログの設計メモ

ブログを作るとき、多くの人はWordPressを選びます。確かに便利ですが、 もっと軽く、もっと自由に、世界観まで含めて表現したいと思い、 今回は静的サイトだけでブログを構築しました。

今回使った構成

使用したのは次の3つだけです。

  • HTML（ページ構造）
  • CSS（和風・高級感ある見た目）
  • JavaScript（モーダル表示・音楽制御）

サーバー側のプログラムやデータベースは使っていません。 GitHub Pagesでそのまま公開できます。

普通のブログとの違い

一般的なブログは、一覧から記事ページへ移動します。 ですが今回は、一覧のカードをクリックすると、その場で和風の閲覧ウィンドウが開く仕様にしました。

これにより、ページ遷移なしで読み進められ、音楽も止まらず、 ブログというより“文集を開く体験”に近づけることができました。

まとめ

技術は派手さより、目的に合っているかが大切です。 軽く、速く、美しく。静的サイトでもここまでできます。

2026.04.24 GitHub Pages

更新したのに反映されない。GitHub Pagesで学んだ“待つ技術”

ブログの文章を書き換えてコミットしたのに、公開サイトには古い内容が残ったまま。 最初はキャッシュか、HTMLのミスか、何か壊してしまったのかと思いました。

しかし GitHub Actions のログを見ると、原因はもっと単純でした。 Pages build and deployment が進行中で、次の公開依頼が順番待ちになっていたのです。

つまり、サイトが壊れたわけではなく、 GitHub側で「今ひとつ前の更新を公開している最中」だっただけでした。

今回学んだこと

  • コミットを連打しない
  • Actionsログを見る癖をつける
  • 反映には少し時間がかかると知る

見た目が静かなサイトでも、裏では多くの処理が動いています。 表面だけで判断せず、仕組みまで理解して整える人でありたいと思いました。

2026.04.25 インフラ

運用現場で“見やすさ”が技術と同じくらい重要な理由

インフラや運用の現場では、「設定が正しいか」だけが評価基準ではありません。 誰が見ても把握しやすいこと、次の人が引き継ぎやすいことも、同じくらい重要です。

正しいだけでは足りない

仕様通りに動く構成であっても、読みづらい設計書や分かりにくい命名は、 後の保守や障害対応を難しくします。

見やすさは事故を減らす

運用現場では、焦っている時ほど見やすい情報が必要になります。 画面、表、ログ、ドキュメントの整え方は、そのまま事故率にも影響します。

技術力とは、ただ動かす力だけではなく、 継続して扱いやすく整える力でもある。そう考えています。
